Part I — Hardware

Hardware Architecture

The system is built around the ESP32 SoC. All peripherals share two buses: I2C (GPIO 21/22) for the DS3231 and LCD, and SPI (GPIO 15/18/19/23) for the SD card. Five tactile buttons and one passive buzzer complete the input/output interface.

Part II — Firmware

Dual-Core Architecture

v3.8 split the firmware across both ESP32 cores using FreeRTOS. Core 1 owns the entire user interface — it runs at a tight 10 ms scan cycle and never makes an HTTP call. Core 0 owns all network operations. A missed button press is now physically impossible regardless of Firebase latency.

Key principle: If Firebase stalls for 20 seconds on Core 0, Core 1 does not notice. Buttons scan at 100 Hz regardless of network state. The FreeRTOS logQueue carries events from Core 1 → Core 0 in under 1 ms.

Part II — Firmware

State Machine

Six boolean flags govern every transition in SENTINEL. The system is event-driven — only BTN5 and timer expiry trigger state changes. Background network events never interrupt user-facing state.

systemActivePasscode entry in progressIdle / standby state

lcdBg() guard: All background status messages check systemActive, isSleeping, AND isComplete before writing to LCD. If any is true, the write is silently dropped. Background processes can never corrupt passcode entry or result screens.
Part II — Firmware

Access Control Logic

A 4-digit code is entered via BTN1–BTN4. On the 4th digit, processCode() runs immediately. The code is matched against the people[] registry. Entry or Exit is determined by the person's current isInside status — last event wins, no pair counting.

Part II — Firmware · v3.9 Primary Feature

Boot Catch-Up System

SENTINEL is powered off each night around 21:00 to protect the hardware. This meant the old 23:55 end-of-day auto-close never fired in practice. Anyone inside at shutdown appeared "still inside" the next morning with no NO-EXIT record.

Real-world problem: Last person leaves ~21:00, powers SENTINEL off. Fixed EOD window at 23:55 never fires. Next morning: isInside=true for everyone who was inside yesterday. v3.9 solves this at boot, before the main task starts.
Three-Layer Boot Catch-Up — Executed before loop() starts
Fully offline-capable: Layer 3 requires only the SD card. Even with a dead RTC coin cell and no WiFi, if the CSV contains a previous day's date, the catch-up fires. No internet dependency for overnight session closure.

report.js — Date calculation (the key fix from v3.6)
// Job fires just after midnight IST. // Always report on the previous day — immune to GitHub runner lag. function getReportDateIST() { const nowUTC = new Date(); const ist = new Date(nowUTC.getTime() + 5.5 * 60 * 60 * 1000); ist.setUTCDate(ist.getUTCDate() - 1); // subtract 1 day unconditionally return `${dd}/${mm}/${yyyy}`; }
Why this works: The cron fires just after midnight IST. "Yesterday IST" is always the fully completed day. GitHub's runner can lag 5–90 minutes — it does not matter because we always subtract exactly 1 day, no hour checks, no conditionals.

Part IV — Project

Version History

v3.3 — v3.5
The Foundation
Dual WiFi, DS3231 RTC with NTP write-back, Firebase REST (replaced Firebase_ESP_Client), SD card logging, nightly GitHub Actions email report, Google Sheets integration, live web dashboard. Seven audited bugs fixed in v3.5 including entry loss, 20-second NTP freeze, and stale fallback date.
v3.6
Report Fixed
GitHub Actions cron was cutting off 35 minutes before end of day, missing late entries. Fixed: cron moved to 35 18 * * * (00:05 IST). report.js now unconditionally subtracts 1 day — immune to runner lag of any duration.
v3.7
Hardware Bugs
Ghost digit buffer: pressing clear did not reset the input buffer — old code merged with new input on next entry. Fixed with atomic clearInput(). Firebase heartbeat HTTP was blocking loop() for up to 20 seconds, making buttons physically unresponsive. Moved to guarded non-blocking path.
v3.8
Dual-Core Architecture
FreeRTOS task split: Core 1 owns UI and buttons exclusively — zero HTTP calls, 10 ms scan cycle, physically impossible to miss a button press. Core 0 handles all network operations. logQueue and lcdQueue pass data between cores in under 1 ms. Firebase stalls on Core 0 are invisible to Core 1.
v3.9 — Current
Boot Catch-Up System
Three-layer boot-time catch-up: DS3231 RTC (primary) → WiFi + NTP (secondary) → CSV date change (offline fallback). On any boot following a session gap ≥10 minutes from ≥21:00, all open sessions receive a NO-EXIT record automatically. End-of-day auto-close now works regardless of whether the device was on at 23:55.
